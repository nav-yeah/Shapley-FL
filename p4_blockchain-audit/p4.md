# P4 — Blockchain Audit Trail for ShapleyFL

Reads the Byzantine-detection results (from P3) and per-round Shapley
scores (from P1), writes each (round, client) entry to an immutable
on-chain ledger, fires `AnomalyFlagged` for flagged rounds, and
demonstrates a reward-eligibility enforcement check that actually
rejects clients with a real attack history.

## Platform choice

Ethereum (Solidity) on a **local Hardhat network**, not the real Sepolia
testnet. Same EVM, same contract, same web3 calls as Sepolia — but it
runs in-process in seconds, with no faucet ETH, no RPC provider, and no
network flakiness. The contract is Sepolia-deployable unchanged; see
"Deploying to real Sepolia" below.

## Status: run and verified, with real flagged clients

Deployed, logged, and demoed against `byzantine_detection_results_merged.csv`
(30 clients, 150 rows, 30 of them genuinely flagged) on 2026-07-18. All
numbers below are real measured output.

- Contract deployed to `0x5FbDB2315678afecb367f032d93F642f64180aa3`
  (local Hardhat network)
- 150 entries logged (30 clients x 5 rounds)
- **30 of 150 rows flagged** — `AnomalyFlagged` fired 30 times, across
  9 distinct clients (9, 10, 13, 16, 18, 24, 25, 28, 29)
- **Total gas used: 30,406,220**
- **Average gas per entry: 202,708**
- Enforcement demo: **21 of 30 clients reward-eligible, 9 rejected**
  — the first real demonstration of the rejection path working

## Data pipeline change — what we changed in `p3_byzantine_detection.py` and why

The original script produced `byzantine_detection_results.csv` by running
the detector on `shapley_scores.csv` — P1's clean 30-client run, which
has no injected attacks. That file is legitimately all-zero
(`flagged_status=0` on every row): there's nothing to flag in clean data.

The script separately evaluates the detector against
`scenario_4_noisy_labels.pkl` (10 clients, with real injected label-flip
attacks) to compute its accuracy numbers (precision=0.875, recall=0.875)
— but that evaluation was previously only summarized as text in
`byzantine_detection_metrics.txt`, never exported as a CSV.

**We added  an export step** — purely
additive, nothing existing was removed or altered — that:

1. Exports the 10-client scenario_4 evaluation on its own:
   `byzantine_scenario4_flagged.csv` (50 rows).
2. Merges those real, attack-evaluated results into the full 30-client
   baseline: `byzantine_detection_results_merged.csv` (150 rows). The
   10 clients that were actually tested against an attack
   (client IDs 1, 9, 10, 13, 16, 18, 24, 25, 28, 29) get their real
   detector output. The other 20 clients keep `flagged_status=0`
   because they were genuinely never attack-tested in this run — an
   `evaluated_for_attack` column marks this explicitly (1 = real
   attack-tested result, 0 = untested clean baseline, not "tested and
   clean") so this distinction is never lost downstream.

**This is the file P4 now uses.** Original outputs
(`byzantine_detection_results.csv`, `byzantine_detection_metrics.txt`,
`byzantine_detection_summary.png`) are unchanged and still produced by
the same run.

Honest framing for the paper: report "flagged 8 of 10 attack-evaluated
clients" (client 1, severity 0.00 = truth negative, correctly
undetected by the method; client 9, severity 0.00, was a false
positive by both naive and method detectors per the client-level
summary) — not "8 of 30," since only 10 clients were ever exposed to
an actual attack in this run.

### Exact code changes made to `p3_byzantine_detection.py`

Confirmed with the script's author before making this change. Purely
additive — no existing line was removed, edited, or reordered; every
original output and threshold is byte-for-byte the same as before.

**1. Two new output path constants** added near the top, next to the
existing `OUT_CSV` / `OUT_TXT` / `OUT_PNG`:
```python
OUT_SCENARIO_CSV = Path("byzantine_scenario4_flagged.csv")
OUT_MERGED_CSV = Path("byzantine_detection_results_merged.csv")
```

**2. One new block added inside `main()`**, immediately after the
existing `plot_severity_summary(severity_df)` call (nothing before that
line was touched):

- Exports the 10-client scenario_4 evaluation (`metrics_df`, which
  already existed in the original script but was previously only
  summarized as text) as its own CSV, `byzantine_scenario4_flagged.csv`
  — 50 rows, with `shapley_value` merged back in from `scenario_df`
  since the detector's feature output doesn't carry it.
- Builds `byzantine_detection_results_merged.csv` — the full 150-row,
  30-client baseline (`deliverable_df`, also already existed) with the
  10 scenario_4 clients' real detector results overlaid on top, plus a
  new `evaluated_for_attack` column (1 = real attack-tested result for
  that row, 0 = untested clean baseline).

**3. Docstring and end-of-run summary text** updated to mention the two
new output files and their row/flag counts, matching the style of the
existing summary lines.

No changes to: `WINDOW_SIZE`, `MIN_SUSTAINED_RATIO`, `TEMPORAL_Z_THRESHOLD`,
`SLOPE_DENOM_FLOOR`, `SEVERITY_BY_POSITION`, `apply_detector()`,
`calibrate_thresholds()`, `run_scenario_rounds()`, or any of the
precision/recall/F1 calculation logic — the detector itself is
untouched, only what gets exported afterward.

## Setup

```bash
npm install
npx hardhat compile
```

## Run order

Open two terminals.

**Terminal 1 — start the local chain (leave running):**
```bash
npx hardhat node
```

**Terminal 2 — deploy, log, and demo:**
```powershell
$env:SHAPLEY_CSV="..\shapley_scores.csv"
$env:BYZ_CSV="..\byzantine_detection_results_merged.csv"

npm run deploy    # deploys ShapleyAudit.sol, writes deployed-address.json
npm run log        # reads both CSVs, logs all 150 entries, writes gas_report.json
npm run demo       # runs reward-eligibility check, writes enforcement_report.json
```

`logResults.js` and `enforcement_demo.js` both read the client list and
flag data directly from whichever file `BYZ_CSV` points at, so this
works unchanged for the 150-row merged file, the 50-row scenario-only
file, or the original all-clean 150-row file — no code changes needed
between runs, only re-pointing the env var and restarting the chain
(`npx hardhat node`) for a clean contract, since `logScore()` rejects
re-logging the same round+client pair.

## Outputs

- `deployed-address.json` — contract address on the local chain
- `gas_report.json` — total + average gas per logged entry, per-row breakdown
- `enforcement_report.json` — per-client flagged/clean counts and eligibility

## Gas cost note

Measured directly from `npm run log` on the local Hardhat network,
150 entries logged (30 clients x 5 rounds, merged dataset with 30
flagged rows):

- **Average gas per entry: 202,708**
- **Total gas used: 30,406,220**

(An earlier run against the all-clean 150-row file measured 198,615 gas
average per entry — the small increase here reflects the extra
`AnomalyFlagged` event emissions and `flaggedCount` writes for flagged
rows, which cost slightly more than the clean-row path.)

Full per-transaction breakdown is in `gas_report.json`.

## Deploying to real Sepolia (optional, for final submission)

1. Get a free RPC URL from Infura or Alchemy.
2. Get Sepolia test ETH from a faucet (e.g. `sepoliafaucet.com`) into a
   throwaway test account — never use a real wallet's private key here.
3. Uncomment the `sepolia` network block in `hardhat.config.js`.
4. Set env vars:
   ```bash
   export SEPOLIA_RPC_URL="https://sepolia.infura.io/v3/YOUR_KEY"
   export SEPOLIA_PRIVATE_KEY="0xYOUR_TEST_ACCOUNT_PRIVATE_KEY"
   ```
5. Re-run the same three commands with `--network sepolia` instead of
   `--network localhost`.

## Files

- `contracts/ShapleyAudit.sol` — the ledger contract (schema: round,
  client_id, shapley_value, flagged_status, hash, timestamp, block number)
- `scripts/deploy.js` — deploys the contract
- `scripts/logResults.js` — reads both CSVs, joins on (round, client_id),
  writes each entry on-chain, tracks gas; driven entirely by whatever
  rows are present in `BYZ_CSV`
- `scripts/enforcement_demo.js` — reward-eligibility check; derives the
  client ID list and clean-round requirement from `BYZ_CSV` rather than
  a hardcoded count, so it works correctly regardless of dataset size
- `deployed-address.json`, `gas_report.json`, `enforcement_report.json` —
  proof-of-run artifacts from the verified run above; commit these
  alongside the code
